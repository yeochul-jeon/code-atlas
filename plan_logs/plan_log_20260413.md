## 오늘의 의사결정 로그

### 1. 핵심 문제 정의
- **결정**: 매 Claude Code 대화마다 동일 코드베이스를 처음부터 재분석하는 비효율이 핵심 문제
- **배경**: Serena MCP는 라이브 심볼 탐색·편집은 가능하지만, 영속적 캐시가 없어 대화 시작마다 재파싱 필요

### 2. 인덱싱 대상
- **결정**: `/Users/cjenm/cjenm/platform/` 하위 9개 Java/Kotlin 프로젝트
- **주요 프로젝트**:
  - `cjos-hexa-arch-sample` — 소규모, 프로토타이핑 우선 대상
  - `cjos-commons` — 중규모, 공통 라이브러리
  - `bts` — 대규모 (36K+ 파일), 성능 검증 대상

### 3. 구현 언어
- **결정**: TypeScript (Node.js)
- **근거**: MCP SDK 공식 지원, Claude Code 생태계와 자연스러운 통합, tree-sitter WASM 바인딩 활용 가능

### 4. 파싱 전략
- **결정**: Hybrid — tree-sitter WASM (필수) + LSP 선택적 보강
- **근거**: `bts` 36K 파일 규모상 LSP 단독 사용은 인덱싱 시간 과다. tree-sitter로 90% 커버 후 필요 시 LSP로 타입 해석 보강
- **기각된 대안**:
  - Tree-sitter Only: 타입 해석/상속 관계 불가
  - LSP Only: jdtls 기동 시간 30-60초/프로젝트, 대규모 처리 부적합

### 5. 저장소
- **결정**: SQLite + FTS5 (전문 검색 포함)
- **근거**: 36K 심볼 처리에 JSON 선형 스캔 불적합, FTS5로 심볼명 검색 밀리초 처리, 단일 파일로 관리 단순
- **미결정 → 확정**: 초기 미정이었으나 SQLite+FTS5로 확정

### 6. 인덱스 갱신 방식
- **결정**: CLI 수동 실행 (`codeatlas index <path>`)
- **기각된 대안**: File watcher 자동 갱신 (데몬 프로세스 필요), Git hook 기반 (설정 복잡)

### 7. Claude 연동 방식
- **결정**: MCP Server (stdio 기본, HTTP 선택)
- **근거**: Claude Code 네이티브 통합 방식

### 8. AI 요약 생성
- **결정**: Lazy 생성 — 첫 `get_file_summary` MCP 호출 시 생성, 이후 캐시 반환
- **근거**: 36K 파일 일괄 요약 시 초기 API 비용 과다. 실제 조회 파일만 요약

### 9. 벡터 임베딩 검색
- **결정**: Phase 1에서 제외, Phase 4 확장으로 연기
- **근거**: 이전 대화(claude.ai)에서 핵심 기능으로 논의되었으나, FTS5로 1차 충분, 복잡도 최소화 우선

### 10. 데드 코드 검출 (신규 추가)
- **결정**: Phase 2에 포함
- **방식**: `references` 테이블에서 참조가 없는 심볼 탐색
- **Spring/Kotlin 제외 규칙**: `@RestController`, `@Service`, `@Component`, `@Repository`, `@Bean`, `main()`, `@Override`, `public static final` 상수
- **MCP 도구**: `find_dead_code`, `get_symbol_references` 추가

### 11. 계획 로그 관리
- **결정**: `/plan_logs/plan_log_{yyyymmdd}.md` 형식으로 프로젝트 내 보관
- **목적**: 의사결정 이력, 대화 요약, 설계 변경 추적

### 13. 벡터 DB 선택: LanceDB (신규 결정)
- **결정**: Phase 5 벡터 검색에 LanceDB 사용
- **근거**: 임베디드 (서버 불필요), TypeScript 네이티브, 파일 기반 (`.codeatlas/vectors/`), 초보자 친화적 API
- **기각 대안**: ChromaDB (Python 중심), Qdrant (서버 필요), Pinecone (클라우드 전용·비용)
- **추가 MCP 도구**: `semantic_search`

### 14. 그래프 DB 선택: Kuzu (신규 결정)
- **결정**: Phase 6 그래프 분석에 Kuzu 사용
- **근거**: 임베디드 (서버 불필요), Node.js 공식 바인딩, Cypher 쿼리 언어 (Neo4j와 동일하여 학습 자료 풍부), 파일 기반 (`.codeatlas/graph/`)
- **기각 대안**: Neo4j (서버 필요, JVM), ArangoDB (서버 필요), TigerGraph (엔터프라이즈)
- **추가 MCP 도구**: `get_impact_analysis`, `find_circular_deps`
- **Phase 재조정**: Phase 5 → LanceDB, Phase 6 → Kuzu (기존 Phase 5에서 분리)

### 12. Serena 대체 — 쓰기 도구 추가 (설계 변경)
- **배경**: Serena는 per-project LSP(jdtls) 기동 구조 → 프로젝트 전환 시마다 30-60초 재기동, 멀티 프로젝트 동시 지원 불가
- **결정**: CodeAtlas에 쓰기 도구 6개 추가하여 Serena 대체
  - `read_symbol_body`, `read_file_range` (읽기 보강)
  - `replace_symbol_body`, `insert_after_symbol`, `insert_before_symbol`, `rename_symbol` (편집)
- **구현 방식**: tree-sitter로 심볼 위치 검증 → 파일 직접 I/O (원자적 쓰기) → 자동 재인덱싱
- **제한**: `rename_symbol`은 텍스트 기반 — 동적 디스패치·리플렉션 사용처 탐지 불가
- **Phase 재조정**:
  - Phase 1: 읽기 도구 10개 (기존 8 + read_symbol_body, read_file_range)
  - Phase 2: 쓰기 도구 4개 (Serena 대체) ← 신규
  - Phase 3: 데드 코드 검출 (기존 Phase 2)
  - Phase 4: AI 요약 (기존 Phase 3)
  - Phase 5: 벡터 검색 (기존 Phase 4)
- **MCP 도구 총계**: 10개 → 16개로 증가
