### 구현 내용
- **신규 파일**: `src/cli/dead-code.ts` — `deadCodeAction` 핸들러 분리

- **테스트**: 7개 신규 (TDD, Red→Green→Refactor)

### 의사결정
- **deadCodeAction 분리**: CLI 핸들러를 `dead-code.ts`로 분리 → 단위 테스트 가능하도록 설계
- **project 미지정 시 전체 스캔**: 인자 없으면 모든 등록 프로젝트 대상
- **`--kind` 옵션**: class|interface|enum|method|field 필터링 지원

### 의사결정 기록

#### 1. DEFAULT_EXTENSIONS 확장
- **결정**: `['.java']` → `['.java', '.js', '.ts', '.jsx', '.tsx', '.vue', '.xml']`
- **근거**: 백엔드(JPA, MyBatis XML) + 프론트엔드(Node.js, Vue.js) 모노레포 지원
- **단, 코틀린(`.kt`)은 현재 비활성화**: 파서가 Java 전용(`parser.ts`에서 `.java` 외 `null` 반환). 추후 활성화 시 tree-sitter-kotlin 파서 연결 필요

#### 2. picomatch @types 미존재 문제
- **문제**: picomatch v4에 `@types/picomatch` 없음
- **해결**: `// @ts-ignore` 주석으로 import 처리
- **기각 대안**: `minimatch` 교체 (불필요한 의존성 추가)

#### 3. annotations 교체 vs append
- **기본값**: `replace_annotations: false` (append 모드)
- **이유**: 기존 Spring 제외 규칙 12개를 실수로 덮어쓰는 사고 방지
- `replace_annotations: true` 명시 시에만 완전 교체

#### 4. `serve` 명령 modelOverride 미연결
- **현재 상태**: `src/cli/index.ts`의 `serve` 명령이 `modelOverride` 없이 호출
- **판단**: MCP 서버가 다중 프로젝트를 서비스하므로, 단일 모델 지정보다는 `find_dead_code` 호출 시 프로젝트별 config 로드가 더 적합
- **현재**: `server.ts`에서 `find_dead_code` 실행 시 `loadConfig(p.root_path)` 호출하여 per-project 처리 중

#### 5. graceful extension 처리
- `collectFiles`는 config에 명시된 모든 확장자 수집
- 파서가 `null`을 반환하면 파일 레코드만 생성, 심볼 없음 — 오류 없음
- 향후 JS/TS/Vue tree-sitter 파서 추가 시 하위 호환
